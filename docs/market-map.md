# Карта рынка

## Проблема

Пользователь локальных LLM обычно проходит одну и ту же цепочку вопросов:

1. Какое у меня железо на самом деле?
2. Какие модели на нём вообще поместятся?
3. Какие кванты и размер контекста имеют смысл?
4. Какая из совместимых моделей лучше под мой use case?
5. Как её скачать, запустить и проверить в реальной работе?

Из этого вытекают 3 продуктовых слоя.

## 1. Fit estimators

Продукты, которые отвечают на вопрос: `что я могу запустить на своём железе?`

### llmfit

- Тип: open-source CLI/TUI
- Роль: самый прямой инструмент под задачу hardware-fit + recommendation
- Сильные стороны:
  - локальный hardware detection
  - ranking по `quality`, `speed`, `fit`, `context`
  - поддержка quantization / providers / plan mode
- Ограничения:
  - больше ориентирован на технического пользователя
  - UX ближе к power-user tooling, чем к массовому продукту

### CanIRun.ai

- Тип: web product
- Роль: browser-first проверка совместимости моделей
- Сильные стороны:
  - очень низкий порог входа
  - хороший обучающий слой вокруг VRAM / params / quantization
  - мгновенный onboarding
- Ограничения:
  - зависит от browser APIs
  - оценки по определению грубее, чем у локального системного детекта

### LocalLLM.run

- Тип: web product / discovery portal
- Роль: совместимость + каталог + сравнение моделей
- Сильные стороны:
  - соединяет fit-check и модельный каталог
  - добавляет compare, leaderboard, guides
  - ближе к productized discovery platform
- Ограничения:
  - не так “локально-точен”, как host-native utility
  - может смещаться в сторону контентного каталога

## 2. Benchmark / evidence layer

Продукты, которые отвечают на вопрос: `ок, модель помещается, но как она реально бежит?`

### LocalScore

- Тип: open benchmark + data layer
- Роль: измерить и показать реальную производительность на конкретном железе
- Сильные стороны:
  - фокус на фактических метриках
  - полезен как доверительный слой поверх fit-оценок
  - даёт базу для ranking по реальному UX, а не по теории
- Ограничения:
  - это не полноценный end-user recommender
  - покрытие железа и сценариев всегда ограничено собранными запусками

### Другие слабые представители слоя

- простые `VRAM calculators`
- нишевые калькуляторы под GGUF

Проблема большинства таких инструментов:

- мало контекста про реальную скорость
- слабая модель use-case ranking
- низкая зрелость и маленькая экосистема

## 3. Execution platforms

Продукты, которые отвечают на вопрос: `я уже выбрал модель, где и как мне с ней работать?`

### Jan

- Тип: open-source desktop app
- Роль: локальное использование модели после выбора
- Сильные стороны:
  - хороший desktop UX
  - мост между model selection и actual usage
  - offline-first позиционирование
- Ограничения:
  - recommendation не является core-product

### GPT4All

- Тип: desktop / local LLM platform
- Роль: запуск локальных моделей и базовый модельный каталог
- Сильные стороны:
  - сильный бренд и зрелость
  - большой охват пользователей локальных моделей
- Ограничения:
  - не решает глубоко задачу hardware-fit intelligence

### LocalAI

- Тип: self-hosted local inference platform
- Роль: OpenAI-compatible backend для локального inference
- Сильные стороны:
  - отличная backend-платформа
  - сильный self-hosted angle
- Ограничения:
  - это execution/server layer, а не recommender

### Ollama

- Тип: runtime / distribution layer
- Роль: де-факто базовый слой локального запуска моделей
- Сильные стороны:
  - огромная экосистема
  - very strong developer mindshare
- Ограничения:
  - пользователь всё ещё сам решает, что именно ему запускать

## Рыночная логика

Сегодня есть разрыв между тремя видами знания:

1. `Hardware knowledge`
   Сколько у пользователя реально доступной RAM / VRAM / CPU / bandwidth.

2. `Model knowledge`
   Какие модели, кванты и размеры контекста существуют и что они дают.

3. `Outcome knowledge`
   Что у пользователя реально получится: скорость, latency, UX и пригодность под конкретную задачу.

Сильный продукт в этой категории должен собрать все 3 слоя в одну систему рекомендаций.

## Наиболее интересная продуктовая ниша

Самая интересная незакрытая зона выглядит так:

`host-native fit estimator + benchmark evidence + recommendation engine + one-click handoff в runtime`

То есть не просто “модель помещается”, а:

- почему именно она рекомендована
- что будет по скорости и UX
- какой quant выбрать
- какой контекст безопасен
- куда нажать, чтобы сразу запустить через Ollama / Jan / llama.cpp
