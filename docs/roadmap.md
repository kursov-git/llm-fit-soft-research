# Практический Roadmap Исследования

## Цель

Сделать из набора наблюдений рабочую исследовательскую базу, которая помогает быстро принимать продуктовые решения, а не просто хранить ссылки на аналоги.

## Результат, к которому ведёт roadmap

На выходе должно быть понятно:

1. какой именно user job стоит брать в фокус
2. чем будущий продукт отличался бы от ближайших аналогов
3. какой MVP имел бы смысл проверять первым
4. какие данные и сигналы нужно собирать для следующего шага

## Принципы

- не уходить в build mode
- не проектировать большую платформу заранее
- держать фокус на практической применимости
- каждую гипотезу переводить в проверяемое утверждение

## Stage 1. Stabilize The Research Base

### Задача

Собрать и зафиксировать базовую карту рынка в одном месте.

### Что уже сделано

- выделены основные рыночные слои
- выбран top-5 ближайших аналогов
- составлена сравнительная матрица
- зафиксированы первичные продуктовые гипотезы

### Критерий завершения

Команда может быстро открыть репозиторий и понять:

- кто прямые конкуренты
- кто только смежные игроки
- какие рыночные гэпы выглядят реальными

## Stage 2. Sharpen The Wedge

### Задача

Сузить тему до одного конкретного и защищаемого user job.

### Варианты wedge

1. `Hardware-fit recommender`
   Лучшие модели и кванты под конкретное железо и use case.

2. `Benchmark-backed advisor`
   Рекомендации, усиленные observed performance на похожем железе.

3. `Runtime handoff layer`
   Интеллектуальный выбор + one-click handoff в Ollama / llama.cpp / Jan.

4. `Context-safe planner`
   Рекомендации по реалистичному контексту, а не только по theoretical max.

### Практический приоритет

Если держаться прагматичной линии, первым стоит углублять:

1. `Hardware-fit recommender`
2. `Benchmark-backed advisor`
3. `Runtime handoff layer`

### Критерий завершения

Есть одно короткое positioning statement формата:

> Мы помогаем X выбрать Y под Z ограничение лучше, чем A/B/C.

## Stage 3. Convert Hypotheses Into Tests

### Задача

Перевести текущие гипотезы из идей в набор проверок.

### Что нужно сделать

1. Для каждой гипотезы выписать:
   - сигнал подтверждения
   - сигнал опровержения
   - минимальный артефакт проверки

2. Разделить гипотезы на:
   - value hypotheses
   - trust hypotheses
   - distribution hypotheses

3. Определить, какие из них можно проверить:
   - руками
   - через лендинг / форму
   - через lightweight prototype

### Критерий завершения

У каждой top-priority гипотезы есть:

- owner question
- test method
- success signal
- next action

## Stage 4. Define The Research MVP

### Задача

Описать минимальный MVP не как software platform, а как testable recommendation artifact.

### Что должно быть определено

1. Целевая аудитория первого прохода
2. Какой вход получает MVP
3. Какой выход он возвращает
4. За счёт чего пользователь понимает, что рекомендации можно доверять
5. Как происходит handoff в существующий runtime

### Критерий завершения

Появляется компактный документ, по которому можно:

- обсудить MVP без абстракций
- оценить scope
- при желании собрать прототип позже

## Stage 5. Build The Evidence Layer

### Задача

Понять, без каких данных рекомендации будут слабыми или недостоверными.

### Минимально нужные данные

1. hardware profile
2. model metadata
3. quantization metadata
4. observed performance
5. user-perceived usefulness

### Практический вопрос

Что из этого можно сначала собрать вручную, а что потребует системы?

### Критерий завершения

Есть список данных, разделённый на:

- available now
- can be manually curated
- requires infrastructure

## Stage 6. Prepare Decision Memo

### Задача

Свести исследование к практическому решению: идти дальше или нет.

### На выходе нужен memo с ответами

1. Есть ли живая ниша между calculator и runtime?
2. Какой wedge наиболее реалистичен?
3. Какой MVP минимален и полезен?
4. Где у идеи потенциальный moat?
5. Что было бы next step, если бы проект продолжался?

## Suggested Immediate Next Steps

Если продолжать исследование, а не разработку, то следующие шаги самые разумные:

1. Сделать отдельный `positioning.md` с 3-5 формулировками продуктового позиционирования.
2. Добавить `user-jobs.md` с сегментами пользователей: solo power user, engineer, AI tinkerer, small team, infra advisor.
3. Добавить `evidence-model.md` с тем, какие данные вообще нужны для trustworthy recommendations.
4. Подготовить `decision-memo.md`, когда накопится достаточно материала.
